# PRODUCT_SCOPE（高校助手能力边界说明）

本文档用于明确：

- 当前版本**已经实现**什么
- 当前版本**暂不承诺**什么
- 面试展示时应如何保证叙事与事实一致

---

## 1. 产品目标（当前版本）

在高校学习场景中，提供一个可闭环的 AI 学习助手：

1. 资料上传与管理
2. 资料分析与学习建议
3. 基于资料的问答
4. 学习测评与报告导出

---

## 2. 已实现能力（Fact）

## 2.1 资料管理

- 上传资料
- 资料列表/详情/删除/下载
- 分类、搜索、统计
- 向量化状态查看与手动重试

对应模块：`modules/course`

## 2.2 资料分析

- 获取课程资料分析结果接口
- 支持分析结果在前端展示（摘要/标签/难度/路径）
- 支持分析报告导出

对应模块：`modules/course`、`modules/student`

## 2.3 问答与会话

- 基于资料的问答接口
- SSE 流式输出
- 会话创建、列表、详情、标题更新、置顶、删除

对应模块：`modules/course/TutorChatController`

## 2.4 测评

- 创建测评会话
- 获取题目、提交答案、暂存答案、提前交卷
- 获取测评详情与报告
- 导出测评 PDF

对应模块：`modules/tutoring`

## 2.5 工程能力

- Redis Stream 异步任务机制（含重试）
- 限流机制（`@RateLimit`）
- 统一错误码与异常返回

---

## 3. 暂不承诺能力（Not in scope for now）

以下能力不建议在面试中描述为“已完成”：

- 完整多租户权限体系
- 完整运营后台与审计体系
- 高并发压测后的 SLA 承诺
- 复杂课程排课/教务系统对接
- 完整实时协作（多人在线编辑类）

可以描述为“规划中/下一步”。

---

## 4. 当前重构状态说明

当前仓库处于重构收口阶段，主要工作项：

1. 术语统一：旧语义（简历/面试）向高校学习语义统一
2. 前端类型契约修正：history/tutoring 页面与 API 对齐
3. 演示链路稳定化：保证端到端可重复演示

---

## 5. 面试叙事建议（务实版）

建议你按以下方式表达，既专业又真实：

1. 我重点做了异步任务与 RAG 的工程落地，而不是只做页面 Demo。
2. 当前功能闭环已可跑通：上传→分析→问答→测评→导出。
3. 项目仍在持续重构，正在做术语一致性和前端契约收口。

这样既能体现工程深度，也不会被追问时陷入事实不一致。
